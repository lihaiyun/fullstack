Testing Your REST API with REST Client

VS Code Extension Student Guide - Tutorial API on http://localhost:3001

Before You Start
  • 1 Install the REST Client extension. Open VS Code, click the Extensions icon (Ctrl+Shift+X), search for REST Client by Huachao Mao, and click Install.
  • 2 Start the server. Open a terminal in the server/ folder and run npm run dev. You should see Server running on http://localhost:3001.
  • 3 Open the .http file. In the VS Code Explorer, navigate to server/test/tutorial.http and open it. You will see Send Request links appear above each request.
✓ Any file with a .http or .rest extension is recognised by the extension automatically - no configuration needed.
How to Send a Request

Each block in the .http file is one request. You can run any request individually - you do not have to run them in order.

  • 1 Click the Send Request link that appears directly above the request line you want to run.
  • 2 A new panel opens on the right showing the status code, response headers, and the response body.
  • 3 The response body is automatically formatted as JSON - you can collapse and expand objects by clicking the arrows on the left.
ⓘ Requests are separated by ### lines. Everything between two ### markers is treated as a single request. Do not remove these separators.
Variables in the .http File

Variables defined with @name = value at the top of the file are reused throughout. This means you only need to change one place to update the server URL or the ID you are testing.

Variables declared in tutorial.http
@baseUrl = http://localhost:3001
@tutorialId = 1

A variable is referenced anywhere in the file using double curly braces:

GET {{baseUrl}}/tutorial/{{tutorialId}}
✓ After you create a tutorial and get back its id, update @tutorialId at the top of the file to that value. All GET, PUT, and DELETE requests that reference {{tutorialId}} will then use the correct ID.

1 - Create a Tutorial (POST)
POST

Create a new tutorial

/tutorial

The request line specifies the method and URL. The Content-Type header tells the server the body is JSON. A blank line separates the headers from the body.

Request in tutorial.http
### Create tutorial
POST {{baseUrl}}/tutorial
Content-Type: application/json

{
    "title": "Learn SQLite",
    "description": "A beginner guide to using SQLite with Sequelize ORM."
}
Expected Response
{ "id": 1, "title": "Learn SQLite", "description": "A beginner guide to using SQLite with Sequelize ORM.", "createdAt": "2026-05-02T04:00:00.000Z", "updatedAt": "2026-05-02T04:00:00.000Z" }
⚠ Validation rules: title must be 3-100 characters and description must be 3-500 characters. Both are required. A missing or too-short field returns 400 Bad Request with an errors array.
✓ Note the id in the response. Update @tutorialId at the top of the file to this value before running the GET, PUT, or DELETE requests below.
200 OK 400 Bad Request
POST

Create tutorial - validation error

/tutorial

This example intentionally triggers a validation error so you can see what the error response looks like.

Request in tutorial.http
### Create tutorial - validation error (title too short)
POST {{baseUrl}}/tutorial
Content-Type: application/json

{
    "title": "Hi",
    "description": "Too short title."
}

{ "errors": [ "title must be at least 3 characters" ] }

2 - Get All Tutorials (GET)
GET

List all tutorials

/tutorial

A plain GET request with no body. The server returns a JSON array of every tutorial, ordered newest first.

Request in tutorial.http
### Get all tutorials
GET {{baseUrl}}/tutorial
Expected Response
[ { "id": 1, "title": "Learn SQLite", "description": "A beginner guide to using SQLite with Sequelize ORM.", "createdAt": "2026-05-02T04:00:00.000Z", "updatedAt": "2026-05-02T04:00:00.000Z" } ]
200 OK
GET

Search tutorials by keyword

/tutorial?search=...

Add a search query parameter to filter results. The server searches both title and description fields.

Request in tutorial.http
### Get all tutorials with search
GET {{baseUrl}}/tutorial?search=sqlite
Expected Response
[ { "id": 1, "title": "Learn SQLite", "description": "A beginner guide to using SQLite with Sequelize ORM." } ]
ⓘ If no tutorials match the keyword, the server returns an empty array [] - not a 404 error.
200 OK

4 - Update a Tutorial (PUT)
PUT

Update an existing tutorial

/tutorial/:id

Both title and description are optional in the body. Omit one to leave it unchanged on the server.

Full update - request in tutorial.http
### Update tutorial
PUT {{baseUrl}}/tutorial/{{tutorialId}}
Content-Type: application/json

{
    "title": "Learn SQLite - Updated",
    "description": "An updated guide to using SQLite with Sequelize ORM."
}
or
Partial update (title only) - request in tutorial.http

5 - Delete a Tutorial (DELETE)
DELETE

Delete a tutorial

/tutorial/:id

No body needed. The server returns 200 OK when the record is successfully deleted.

Request in tutorial.http
### Delete tutorial
DELETE {{baseUrl}}/tutorial/{{tutorialId}}
Expected Response
HTTP/1.1 200 OK
✓ After deleting, click Send Request on the Get tutorial by id request. You should get a 404, confirming the record is gone.
200 OK 404 Not Found

Reading the Response Panel

When you click Send Request, a panel opens on the right side of VS Code showing three things:

  • 1 Status line - e.g. HTTP/1.1 200 OK. The number tells you whether the request succeeded. 2xx = success, 4xx = client error.
  • 2 Response headers - metadata like Content-Type and Date. Usually you can ignore these.
  • 3 Response body - the JSON data returned by the server. REST Client formats it with colour and indentation automatically.
ⓘ Common status codes you will see: 200 success, 400 bad request (validation failed - check the errors array in the body), 404 record not found.
Useful Tips
  • ✓ Run requests in order when testing the full flow - POST first to create a record, then GET to read it, PUT to update, DELETE to remove.
  • ✓ Keep the server terminal visible. Error messages printed by the server (e.g. Sequelize errors) appear there, not in VS Code.
  • ✓ Check the status code first. If something looks wrong, the status line tells you whether the problem is in your request (4xx) or on the server (5xx).
  • ✓ You can add your own requests. Copy any block, paste it below a ### separator, and modify the body or URL. Your additions are saved in the file.
